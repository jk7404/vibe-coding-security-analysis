# **Evaluating the Security Degradation of Vibe Coding vs. Structured Prompting in Profile Picture and Document Upload Handlers**

## Jaehyeon Kim, Daewoong Lim, Sihyun Kim, Yongje Jeon

### **Motivation**

The rapid adoption of Large Language Models (LLMs) has popularized "vibe coding"—a paradigm where developers use minimal, conversational prompts to generate functional code rapidly. While efficient, this approach introduces severe, architecturally invisible security risks. When tasked with "innocent-looking" features like profile picture or resume document upload handlers, LLMs frequently assume user inputs are benign. Without a holistic understanding of server boundaries, models routinely save files using user-supplied filenames, opening the door to devastating Path Traversal (CWE-22) vulnerabilities where attackers overwrite critical system files (e.g., ../../../etc/passwd). This topic is highly engaging because it contrasts the semantic over-confidence of AI models in mundane tasks against the strict, deterministic verification provided by static program analysis.

### **Related Work**

Recent empirical research underscores the hidden dangers of AI-assisted coding. A large-scale evaluation of LLM outputs from over 4,000 coding assignments using SonarQube demonstrated that models such as GPT-4o and Claude 3.5 Sonnet consistently inject critically severe issues, explicitly including path traversal vulnerabilities. To detect these complex logic flaws, researchers have increasingly relied on advanced static analysis frameworks like CodeQL, which uses symbolic logic for whole-repository taint tracking to identify CWE-22 Path-Traversal vulnerabilities that standard linters miss.

### **Proposal**

We plan to explore and quantify how different prompting strategies affect the injection rate of Path Traversal vulnerabilities in LLM-generated Python web applications (e.g., Flask/FastAPI backend upload handlers). We will implement four distinct prompting methods:

1. **Unconstrained Vibe Coding:** Minimal context, zero-shot conversational prompting.  
2. **Zero-Shot Chain-of-Thought (CoT):** Prompting the model to output reasoning steps prior to generating the upload logic.  
3. **Plan-and-Solve:** A structured approach requiring the LLM to outline an architectural plan and define security edge cases before writing code.  
4. **Test-Driven Development (TDD):** Providing the LLM with predefined unit tests (including malicious path traversal inputs) that the generated code must pass.

Our four-person team will divide responsibilities as follows:

* **Member 1 (Vibe Prompting Lead):** Design and execute the Unconstrained and CoT prompts to generate the baseline upload handlers.  
* **Member 2 (Structured Prompting Lead):** Design and execute the Plan-and-Solve and TDD prompts.  
* **Member 3 (Static Analysis Lead \- Security):** Configure and execute advanced static analysis using CodeQL to perform inter-procedural taint analysis, tracking unverified file inputs to critical file-saving sinks.  
* **Member 4 (Static Analysis Lead \- Quality):** Configure AST-based tools like Bandit to detect general security anti-patterns and aggregate the final dataset.

### **Experimental Evaluation**

Our work will be validated empirically. We will define three distinct, common file-handling tasks (e.g., a user profile picture uploader, a CSV data importer, and a PDF resume parser). Using a frontier model like GPT-4o, we will generate solutions for these tasks across our four prompting strategies. The generated codebases will be systematically evaluated using Bandit (for AST parsing) and CodeQL (for deep data-flow and taint tracking). We will measure the defect density and explicitly count the occurrences of CWE-22 vulnerabilities across the outputs. The evaluation will validate our hypothesis by comparing the False Discovery Rate and vulnerability severity generated by unconstrained methods versus structured methods.

### **Expectations**

We expect the empirical evaluation to demonstrate that unconstrained vibe coding yields a high rate of severe Path Traversal vulnerabilities, as the LLM will fail to proactively implement path sanitization (e.g., `werkzeug.utils.secure_filename`). Conversely, we expect the structured methodologies—particularly Plan-and-Solve and TDD—to force the model to acknowledge security boundaries, resulting in code that successfully neutralizes malicious inputs and passes CodeQL taint analysis.
