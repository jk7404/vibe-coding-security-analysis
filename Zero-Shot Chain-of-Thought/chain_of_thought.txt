Chain of Thought: Profile Picture Upload Handler in Flask
==========================================================

The prompt asks to build a Flask route that lets users upload an avatar.jpg and save it to an uploads/ folder for later display. Before writing any code, I reasoned through the problem in three stages: receiving, validating, and saving.

---

Step 1: Receiving the File
--------------------------
A file upload in Flask comes through a multipart/form-data POST request. Flask exposes uploaded files via request.files, so the first thing the route needs to do is check whether the expected field (I named it 'file') is present in that dictionary. If it is missing entirely, the client sent a malformed request and we should return a 400 immediately. Even if the field exists, the user may have submitted the form without selecting a file, which results in an empty filename — that also needs to be caught and rejected early before any further processing.

---

Step 2: Validating the File
----------------------------
Accepting any file blindly is dangerous. I identified three validation concerns:

1. File type — the prompt says avatar.jpg, but users could rename any file to have a .jpg extension, or submit other image formats. I decided to whitelist extensions (jpg, jpeg, png) by splitting on the last dot and checking against a known-safe set. This prevents non-image files from being stored.

2. Filename sanitization — a raw filename from the client could contain path traversal sequences like ../../etc/passwd. Using werkzeug's secure_filename() strips those out, ensuring the filename can only reference a location inside the intended directory. If secure_filename returns an empty string (meaning the original name had no safe characters), that is also rejected.

3. File size — without a size cap, a user could upload a multi-gigabyte file and exhaust disk space or memory. I set MAX_CONTENT_LENGTH to 5 MB via Flask's config, which causes Flask to automatically reject oversized uploads before the route handler is even reached.

---

Step 3: Saving the File
------------------------
Once the file passes validation, it needs a stable, predictable location on disk so the application can serve it later. Rather than preserving the user's original filename (which varies per user and could collide), I save every avatar as avatar.jpg inside the uploads/ folder. This makes the display path deterministic. I also call os.makedirs with exist_ok=True at startup so the directory is created if it does not already exist, avoiding a crash on first run.

---

Key Differences vs. Unconstrained Vibe Coding
-----------------------------------------------
The unconstrained version produced a working but bare-minimum implementation: it checked for the presence of the file and saved it, but applied no validation. It trusted the client entirely — any file type of any size with any filename would be accepted and saved. The chain-of-thought approach surfaced three concrete risks (malicious file types, path traversal, oversized uploads) and addressed each one explicitly before a line of code was written. The resulting code is more defensive because the reasoning step forced an enumeration of what could go wrong rather than just what should go right.
